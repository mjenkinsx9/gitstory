# GitStory

**A CLI tool that analyzes any git repository and generates a stunning, self-contained HTML narrative timeline.**

GitStory transforms raw git history into a cinematic visual story. Point it at any repository and it produces a single HTML file with D3.js-powered visualizations, scroll-driven animations, and auto-generated chapter titles that reveal the rhythm, intensity, and human effort behind a codebase.

## Features

- **7 cinematic sections**: Hero, Prologue, The Pulse, The Story, The Cast, The Rhythm, Epilogue
- **D3.js visualizations**: Mirrored area chart (additions/deletions), contributor swimlane chart, GitHub-style activity heatmap
- **Narrative intelligence**: Commits are clustered into chapters using time-gap detection, with creative titles generated from commit patterns ("The Great Refactoring", "Bug Squashing Marathon", "Feature Sprint: authentication")
- **Dark cinematic theme**: `#06060e` background, glassmorphism cards, gradient accents, noise texture overlay, Inter + JetBrains Mono typography
- **Scroll animations**: IntersectionObserver-driven fade-ins, animated stat counters, hover tooltips with backdrop blur
- **Single HTML output**: No server needed. Open the file in any browser.
- **Remote support**: Analyze GitHub repos by URL (clones to temp directory, cleans up automatically)

## Installation

```bash
git clone https://github.com/mjenkinsx9/gitstory.git
cd gitstory
npm install
npm run build
```

## Usage

```bash
# Analyze the current directory
node dist/index.js analyze

# Analyze a specific repository
node dist/index.js analyze /path/to/repo

# Analyze a remote GitHub repository
node dist/index.js analyze https://github.com/user/repo

# Customize output
node dist/index.js analyze . -o my-story.html --title "My Project's Journey" -n 500
```

### CLI Options

```
gitstory analyze [repo-path]

Arguments:
  repo-path              Path to git repository or GitHub URL (default: ".")

Options:
  -o, --output <file>    Output HTML file (default: "gitstory.html")
  -n, --max-commits <n>  Maximum commits to analyze
  --title <string>       Custom title for the story
  -V, --version          Display version
  -h, --help             Display help
```

### Output Sections

| Section | Description |
|---------|-------------|
| **Hero** | Full-viewport intro with repo name, key stats, gradient text, animated floating orbs |
| **Prologue** | Animated stat counters that count up on scroll (commits, contributors, lines added/removed) |
| **The Pulse** | Mirrored D3.js area chart -- additions above centerline (cyan), deletions below (rose), with glow effects and hover tooltips |

## The Build Story

> An entire CLI tool with cinematic HTML output, D3.js visualizations, and 196 tests -- conceived, planned, built, tested, and shipped in under 90 minutes through a six-phase agentic workflow.

### The Problem

Git history is data-rich but human-hostile. Run `git log` on any project and you get a wall of hashes, timestamps, and terse messages. The structure is there -- the sprints and lulls, the solo sessions and team surges, the refactoring campaigns and feature buildouts -- but it is buried in plain text.

GitStory exists to surface that structure. The goal: point a CLI at any git repository and produce a single HTML file that tells the story of how that codebase was built. Not a static report. A cinematic, scroll-driven narrative with D3.js visualizations, animated counters, chapter titles like "The Great Refactoring" generated from actual commit patterns, and a dark theme that makes the data feel like it matters.

The challenge was not just building the tool. It was building it from nothing -- no `package.json`, no `src/` directory, no configuration -- in a single session, using a coordinated team of AI agents.

### The Plan

The build followed Forge's six-phase workflow: **Discuss, Spec, Plan, Execute, Audit, Monitor**. Each phase was handled by specialized agents, and the handoffs between them produced the artifacts that guided the entire build.

**Phase 1: Discuss** -- Two investigator agents worked in parallel. The **issue-investigator** analyzed GitHub Issue #1, extracting 11 acceptance criteria, mapping functional requirements across four domains (git analysis, narrative generation, visual frontend, CLI interface), and flagging 7 open questions about chapter clustering algorithms, remote repo scope, and test coverage expectations. Simultaneously, the **codebase-investigator** surveyed the empty project and produced a structural blueprint: 14 source files to create, 3 npm dependencies to install, 5 integration points to wire together. Both agents assessed their confidence as HIGH -- the issue was exceptionally detailed, leaving little ambiguous.

**Phase 2: Spec** -- The **spec agent** synthesized the investigator reports into a 280-line specification document covering every requirement, data model (6 TypeScript interfaces), programmatic API (9 function signatures), user flows (2 happy paths, 5 error paths, 6 edge cases), and 12 testable acceptance criteria. The **spec-reviewer agent** then audited this document, verifying 28 of 30 requirements were fully covered and finding one blocking issue: the testing plan was underspecified. The spec was amended with a comprehensive testing section mapping unit tests to 6 source areas plus HTML validation and integration tests.

**Phase 3: Plan** -- The **plan agent** decomposed the spec into 19 tasks across 5 waves, with explicit dependency ordering and TDD steps for each task. The **plan-reviewer agent** validated the wave DAG (no circular dependencies, no file conflicts), confirmed 37 of 39 requirements were covered by specific tasks, and noted that 2 edge cases (permission denied errors, large-repo timeout warnings) were not explicitly tasked but would likely be handled naturally. Research findings -- Commander v14 over v15, `moduleResolution: nodenext` with `.js` extensions, `execSync` with 50MB `maxBuffer` -- were all incorporated.

### The Build

Execution used **subagent-driven development**: the orchestrator dispatched each task to a dedicated subagent, which followed the TDD cycle (write failing test, implement, verify, commit) before reporting back.

**Wave 1: Foundation** (commits `9b0cf59`, `9352c6d`) -- Project scaffolding came first. `package.json` with ESM configuration, `tsconfig.json` with strict mode and `nodenext` resolution, `vitest.config.ts` with the test include pattern. Then 6 TypeScript interfaces -- `GitCommit`, `FileChange`, `Chapter`, `ContributorSummary`, `StoryData`, `DailyActivity` -- establishing the data model that every subsequent module would import. The type definitions came with 147 lines of type-level assertion tests to verify the interfaces were correctly shaped.

**Wave 2: Core Logic** (commits `6632bdb` through `46ff105`) -- Five modules built in rapid succession, each with its own test suite. The git log parser (`src/git/analyzer.ts`) uses `execSync` with a structured `--format` string using `COMMIT_START`/`COMMIT_END` delimiters and `--numstat` for file stats, handling edge cases like binary files (`-\t-\t` entries), empty repos, and UTF-8 author names. Remote support (`src/git/remote.ts`) validates GitHub HTTPS URLs via regex and clones to `mkdtempSync` directories with a 2-minute timeout. Chapter clustering (`src/narrative/chapters.ts`) walks sorted commits and breaks on 24-hour gaps, computing intensity as a normalized commits-per-day metric. Title generation (`src/narrative/titles.ts`) applies pattern-matching heuristics: high deletion ratios trigger "The Great Refactoring" style titles, `fix`/`bug` commits trigger "Bug Squashing Marathon", single-directory focus triggers "Feature Sprint: [dir]", and single-author chapters get personalized titles. Statistics computation (`src/utils/stats.ts`) handles contributor aggregation by email and daily activity bucketing.

**Wave 3: HTML Generation** (commits `2fe7b51` through `002f60f`) -- The largest wave, producing 8 commits and over 1,500 lines of template code. The CSS module (`src/report/styles.ts`) alone is 667 lines: dark cinematic theme, glassmorphism with `backdrop-filter: blur`, noise texture overlay via SVG `feTurbulence` filter, responsive breakpoints at 768px and 375px, and fade-in animation classes. Each of the 7 HTML sections is a dedicated TypeScript function returning template literal strings. The D3.js visualization code -- mirrored area charts with `curveMonotoneX` smoothing and gradient fills, contributor swimlane bars with `scaleBand` and `scaleTime`, and a heatmap calendar positioned by week and day-of-week indices -- lives as JavaScript strings inside TypeScript, sacrificing IDE support for the simplicity of zero build-time dependencies. The report generator (`src/report/generator.ts`) assembles everything: HTML head with Google Fonts CDN links, inlined CSS, all 7 section templates, the D3.js CDN script tag, embedded `STORY_DATA` as JSON, and IntersectionObserver initialization for scroll animations and counter triggers.

**Wave 4: CLI Integration** (commit `2a5dc2f`) -- The entry point wired everything together with Commander v14: argument parsing, GitHub URL detection routing, the full analysis pipeline (parse, cluster, title, stats, generate, write), error handling with descriptive messages and exit code 1, and `--max-commits` validation. The shebang line and `bin` entry in `package.json` make it installable as a global command.

**Wave 5: Validation** (commits `479fca2`, `d510c7f`) -- Integration tests create temporary git repositories in `beforeAll` with programmatic commits from multiple authors at varied timestamps, then run the full pipeline and verify the HTML output contains all expected sections and data. HTML validation tests parse the generated output to verify all 7 section IDs, the D3.js CDN tag, Google Fonts links, IntersectionObserver code, tooltip elements with mouseover handlers, responsive media queries for both breakpoints, valid embedded JSON data, and counter animation code. Final verification confirmed `npm run build`, `npx tsc --noEmit`, and all 196 tests passing, plus a smoke test that produced a 77KB HTML file from the GitStory repo itself.

### The Challenges

The build was notably smooth, which is itself worth examining. Three factors contributed:

First, the issue was extraordinarily detailed. GitHub Issue #1 contained section-by-section UI requirements, explicit acceptance criteria, and specific technology mandates. The investigator agents had little to disambiguate, which meant the spec was tight and the plan was actionable from the start.

Second, the research phase resolved potential pitfalls before they became implementation problems. The **researcher agent** discovered that Commander v15 was a pre-release requiring Node.js v22.12+ and recommended v14 instead. It identified that `moduleResolution: nodenext` with `.js` import extensions was required for ESM without a bundler -- a common source of build failures. It calculated that `execSync` with a 50MB `maxBuffer` would handle repos up to approximately 50,000 commits. These findings, all rated HIGH confidence, were incorporated directly into the plan.

Third, the subagent-driven execution model meant each task was self-contained. A subagent implementing the heatmap calendar did not need to know about the chapter clustering algorithm. The type definitions from Wave 1 served as the contract, and every module honored it.

The one structural tension was the D3.js visualization code: JavaScript written as TypeScript template literal strings, with no IDE support, no type checking, and no way to catch errors until the HTML is opened in a browser. The mitigation -- comprehensive HTML validation tests that verify the presence of D3 method calls, gradient definitions, tooltip handlers, and responsive breakpoints via string matching -- is pragmatic rather than elegant. It works, but it is the part of the codebase most likely to break under future changes.

### The Review

Quality assurance ran at two levels. The plan mandated a **two-stage review order** for every task: first spec compliance (does the implementation match SPEC.md requirements?), then code quality (do tests pass, does lint pass, are there secrets or `any` abuse?).

The **spec-reviewer agent** caught that the original specification lacked a testing section despite CONTEXT.md requiring tests across 5 areas. This was fixed before planning began, ensuring the plan included test tasks from the start.

The **plan-reviewer agent** verified the wave dependency DAG had no cycles, confirmed no file conflicts existed between same-wave tasks, validated that all 15 implementation tasks followed TDD steps, and checked that all HIGH-confidence research findings were incorporated. It flagged 2 missing SPEC requirements (permission denied handling, large-repo timeout suggestions) as advisory -- not blocking execution but worth noting.

The final verification (Wave 5, Task 3) ran the complete toolchain: `npm run build` with zero errors, `npx tsc --noEmit` passing strict mode, and all 196 tests green across 17 test files in 1.47 seconds.

The entire build -- from empty repository to merged PR -- completed in under 90 minutes. Sixteen commits on the `feat/build-gitstory` branch tell the story: scaffolding, types, five core logic modules, CSS styles, seven HTML sections, a report generator, a CLI entry point, and two test suites.

What the agentic workflow enabled was not speed alone but *structured speed*. The parallel investigator agents ensured nothing was missed in requirements gathering. And the TDD discipline -- enforced by the plan, not by willpower -- meant that 196 tests existed before the final verification step, not as an afterthought.

The result is a tool that does what it promises: point it at a git repo and it tells you the story of how that code came to be.
